# Workflow: от задачи до коммита

Выстроенный процесс работы над задачами через AI-агентов.

## Этапы

```
Задача → Ресерч → Анализ → Разработка → Тест → Коммит
```

### 1. Ресерч

Запусти `researcher` для изучения контекста:
```
/research <вопрос или описание задачи>
```
Или в основном чате:
```
Запусти researcher: нужно понять как X
```

Researcher работает в фоне. Читает, ищет, не правит файлы.

### 2. Анализ

Запусти `architect` для оценки подхода:
```
/review <описание решения>
```

Architect оценивает trade-offs, указывает на проблемы, предлагает альтернативы. Только читает.

### 3. Разработка

**Небольшая задача** — запусти агента с нужными правами прямо из чата:
```
Запусти backend-агента: реализовать X в сервисе Y
```

**Большая задача или эксперимент** — изолированный worktree:
```bash
./modules/scripts/worktree-add.sh git@github.com:org/service.git task-name main
claude tmp/service-task-name
```
Или через команду:
```
/dev <описание задачи>
```

Основная ветка остаётся чистой. Агент работает независимо.

### 4. Тест

Перед коммитом — убедись что работает:
```bash
# Запусти приложение и проверь happy path вручную
# Прогони тесты
go test ./...        # Go
vitest run           # TypeScript
pytest               # Python
```

### 5. Коммит

lefthook запускает проверки автоматически при `git commit`:
- pre-commit: lint, format, typecheck
- pre-push: tests

Если что-то упало — исправь причину, не пропускай (`--no-verify`).

## Параллельная работа

Независимые задачи можно вести параллельно:

```
Задача A → worktree-A → агент-A →  ↘
                                    merge
Задача B → worktree-B → агент-B →  ↗
```

Ограничение: задачи с зависимостями — последовательно.

## Оркестратор vs агент

| Режим | Когда | Как |
|---|---|---|
| **Оркестратор** | Задача затрагивает несколько сервисов | Основной чат координирует агентов |
| **Изолированный агент** | Большая задача в одном репо | `worktree-add.sh` + отдельный `claude` |
