# memory

Структурированный индекс поверх встроенной памяти Claude Code.

## Контекст: что уже делает Claude Code

Claude Code имеет встроенную трёхслойную память:
- **Session memory** — текущий контекст разговора
- **Project memory** — `CLAUDE.md` в корне репо, читается автоматически
- **Auto-memory** — Claude сам решает что сохранить между сессиями, пишет в `~/.claude/projects/.../memory/`

Этот модуль — **надстройка**, не замена. Встроенный auto-memory хранит что Claude счёл важным. `MEMORY.md` — явный контролируемый индекс: ты решаешь что туда попадает и в каком формате.

## Зачем модуль поверх встроенного

- **Явный контроль** — ты видишь и правишь что помнит Claude, не гадаешь
- **Структура по типам** — user / project / feedback / reference вместо плоского файла
- **Переносимость** — файлы в репо, не в `~/.claude/` — команда видит тот же контекст
- **Обзор** — `MEMORY.md` как оглавление, одним взглядом видно весь накопленный контекст

## Структура

```
modules/memory/
  MEMORY.md        ← индекс всех записей (читается в начале каждой сессии)
  user.md          ← профиль: роль, опыт, предпочтения
  project_*.md     ← контекст проектов: цели, решения, статус
  feedback_*.md    ← что работает, что нет — не повторять ошибок
  reference_*.md   ← где искать: Linear, Jira, Confluence, Grafana
```

## Формат файла

```markdown
---
name: Название
description: Одна строка — используется для поиска релевантных записей
type: user | project | feedback | reference
---

Содержание записи.
**Why:** причина почему это важно.
**How to apply:** как применять в работе.
```

## Правило: что класть, что нет

Класть:
- Архитектурные решения и их причины
- Предпочтения по стилю и процессу (что делать / не делать)
- Контекст проектов который не вытащить из кода
- Ссылки на внешние системы

Не класть:
- То что есть в коде или git-истории
- Временные задачи (для этого todo.md в проекте)
- То что уже в CLAUDE.md
